# ServiceDesk

Система управления заявками (Service Desk) с чатом в реальном времени, ролевой моделью доступа, базой знаний и панелями дашбордов для разных полей пользователей

## Технологический стек
- **Backend**: Python + Flask
- **База данных**: SQLAlchemy + Alembic (миграции)
- **Аутентификация**: Flask-Login
- **Реалтайм**: Flask-SocketIO
- **Формы**: WTForms
- **Frontend**: Jinja2 + JS
- **Дополнительно**: Flask-Migrate, CSRF-защита

## Структура проекта

Проект организован по принципу **Blueprints** (модульная архитектура):

```
├───app - основной пакет приложения
│   ├───blueprints - все маршруты
│   ├───forms - WTForms
│   ├───models - SQLAlchemy-модели
│   ├───services - бизнес-логика
│   ├───static - CSS, JS, шрифты, изображения
│   ├───templates - Jinja2-шаблоны
│   ├───utils - утилиты и декораторы
├───migrations - миграции Alembic
├───packages - используемые пакеты
├───tests - тесты
├───config.py - конфигурация
├───run.py / wsgi.py - точки запуска
```

## 4. Разъяснения по коду

### Models

В папке Models лежат модели баз данных приложения, написанные с помощью SQLAlchemy. Не рекомендуется вносить изменения в модели приложения, поскольку это может напрямую повлиять на работу приложения. 

Если всё же необходимо внести изменения, то алгоритм действий следующий: 
- в необходимом файле ".py" выполняем необходимые изменения. 
- сохраняем файл.
- выполняем миграцию базы данных командой "py -m db migrate".
- обновляем базу данных командой "py -m db update".

### Services 

В папке Services лежат файлы ".py", которые содержат в себе бизнес-логику приложения. Чаще всего, это статические методы, которые возвращают какие-либо данные. Например, статический метод "create_ticket" возвращает данные по заявке, при этом записывая лог в таблицу activity_logs. Эти статические методы могут использоваться во всех файлах ".py" приложения с предварительным импортом самих сервисов. 

Приведем пример: нам необходимо создать лог на метод закрытия заявки в приложении. Для этого мы находим файл, в котором содержится метод закрытия заявки (в нашем случае это resolve_ticket), в начало этого файла прописать импорт: 
```
from app.services.log_service import LogService
```
и вставить в этот метод функцию добавления лога.

## 5. Правила нейминга

### Python
* Функции и методы: ```snake_case()```
* Классы: ```PascalCase```
* Переменные: ```snake_case```
* Константы: ```UPPER_SNAKE_CASE```
* Blueprints: ```ticket_bp = Blueprint('ticket', __name__, url_prefix='/tickets')```

### Шаблоны и JS
* Файлы: ```snake_case.html```
* JS-функции: ```camelCase()```

## 6. Основные возможности
- Создание, просмотр и маршрутизация заявок
- Чат в реальном времени внутри заявки
- Прикрепление файлов к сообщениям
- Ролевая модель: Admin, Classifier, Executor, User
- База знаний (расширенная для исполнителей)
- Поиск по заявкам и статьям
- Уведомления
- Раздельные дашборды для каждой роли

## 7. Важные замечания
- Все изменения моделей БД должны сопровождаться миграцией
- Для продакшена рекомендуется использовать Gunicorn + uWSGI или другой WSGI-сервер + отдельный сервер для Socket.IO (evenlet/gevent)
