# Разбор ошибки «404 Not Found» и неработающего входа (Apache + gunicorn + Kerberos)

Документ объясняет, **почему сайт отдавал 404 / не пускал по домену**, что
исправлено и как теперь диагностировать подобные проблемы.

## Короткий итог

Службы (`servicedesk`, `httpd2`) поднимались корректно, gunicorn слушал
`127.0.0.1:8000`, Apache проксировал на него. Но вход не работал из-за **трёх
несостыковок между конфигом Apache и кодом приложения**:

1. **Имя пользователя бралось не оттуда.** Apache отдаёт имя доменного
   пользователя в HTTP-заголовке `X-Remote-User`
   (`RequestHeader set X-Remote-User "%{REMOTE_USER}s"`), а код читал его из
   WSGI-переменной окружения `REMOTE_USER`. **При проксировании на gunicorn
   `REMOTE_USER` в окружении не появляется** (он есть только у mod_wsgi/mod_cgi).
   Результат: даже успешно прошедший Kerberos пользователь не логинился, и
   Flask-Login отправлял его на `/auth/login`.

2. **Исключения в Apache указывали на несуществующие пути.** Блюпринт `auth`
   имеет префикс `/auth`, то есть вход — это `/auth/login` и `/auth/logout`.
   А в конфиге без авторизации пропускались `/login` и `/logout`. Поэтому
   реальная страница входа `/auth/login` оказывалась **за обязательным
   Kerberos**, и запасной/локальный вход при недоступном домене был невозможен.
   Блок `<Location /auth/login?next=%2F>` не работал вовсе — в `<Location>`
   строку запроса (`?next=...`) указывать нельзя.

3. **Почему именно 404 от gunicorn.** `404 Not Found` с заголовком
   `Server: gunicorn` означает, что ответ сформировало **само приложение**, а не
   Apache. Так бывает, когда запрос дошёл до маршрута, которого в приложении нет
   (например, после правок кода gunicorn **не был перезапущен** и продолжал
   обслуживать старую версию, либо запрашивался путь вроде `/login`, которого в
   приложении нет — есть только `/auth/login`). В актуальном коде `/auth/login`
   существует и отвечает `200`.

Дополнительно: **`journalctl` показывал «no entries»**, потому что gunicorn не
писал журнал в stdout, а `httpd2` пишет в файлы, а не в journald. И смотрелись
**не те лог-файлы** — общий `/var/log/httpd2/error_log` вместо журналов сайта
`/var/log/httpd2/sd_error.log` и `sd_access.log`.

## Что исправлено в репозитории

- `blueprints/auth/routes.py` — функция `get_remote_user()` читает имя в порядке
  `X-Remote-User` → `REMOTE_USER` → `LOGON_USER` и игнорирует пустое значение и
  литерал `(null)` (его Apache подставляет на путях без авторизации).
- `__init__.py` —
  - логи приложения направлены в stdout (видны в `journalctl -u servicedesk`);
  - логируется каждый запрос: метод, путь, код ответа, `X-Remote-User`, IP;
  - обработчики ошибок **404/403/500** возвращают понятную страницу: «ответ
    сформировал Flask/gunicorn», какой путь, какое имя пользователя пришло от
    Apache (пусто → Kerberos/Basic не сработал);
  - добавлен `/healthz` (без авторизации) — проверка живости приложения и БД.
- `config/sites-enabled/servicedesk.conf` и `deploy/servicedesk.conf` —
  исключения исправлены на `^/auth/(login|logout)` через `<LocationMatch>`,
  добавлено исключение `/healthz`, удалён неработавший блок с `?next=`.
- `deploy/servicedesk.service` — у gunicorn добавлены
  `--access-logfile - --error-logfile - --capture-output --log-level info`.
- `config.py` — из кода убран реальный пароль сервисной учётки AD (был
  закоммичен). **Этот пароль скомпрометирован — смените его в домене** и держите
  значения только в `.env`.

## Как теперь диагностировать «404 / не пускает»

1. **Жив ли само приложение:**
   `curl -k https://<сервер>/healthz` — если `{"status":"ok"...}`, то gunicorn и
   приложение работают, проблема в Apache/авторизации.
2. **Кто отдал 404** — по странице ошибки видно «ответ сформировал Flask/gunicorn».
   Если страница обычная Apache — дело в Apache (vhost/ProxyPass).
3. **Логи приложения:** `journalctl -u servicedesk -e` (теперь там есть записи).
4. **Логи именно этого сайта в Apache:** `tail -f /var/log/httpd2/sd_error.log`
   и `/var/log/httpd2/sd_access.log` (а НЕ общий `error_log`/`access_log`).
5. **После любых правок кода** обязательно `systemctl restart servicedesk`
   (gunicorn не перечитывает код сам), после правок конфига Apache —
   `apachectl configtest && systemctl reload httpd2`.

## Сценарии входа (после исправления)

- Доменный ПК, билет Kerberos есть → вход автоматически, без пароля.
- Kerberos не сработал → браузер спросит логин/пароль → **доменная** учётка
  (Apache проверит через KDC, `GssapiBasicAuth On`).
- Домен/KDC недоступен → открыть `https://<сервер>/auth/login` и войти
  **локальной** учёткой (запасной админ/тестовые пользователи).
