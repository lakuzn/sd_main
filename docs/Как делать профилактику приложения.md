# Профилактика запуска и устранение ошибок (ServiceDesk: Flask + gunicorn + Apache)

Документ для того, чтобы запуск приложения «получился с первого раза», а если
что-то пошло не так — чтобы **за минуты** понять, на каком слое проблема.
Здесь же — разбор того, почему `curl -I` к gunicorn показывал `200 OK`, а сайт
всё равно отдавал `404`.

## 1. Самая частая причина «не запускается»: НЕПРАВИЛЬНАЯ РАСКЛАДКА ФАЙЛОВ

Приложение — это **пакет Python с именем `app`**. Поэтому файлы нельзя сваливать
«в кучу». Раскладка строго такая:

```
/opt/servicedesk/                 ← рабочая папка (WorkingDirectory, cwd gunicorn)
├── run.py                        ← точка входа (gunicorn run:app)
├── config.py                     ← читается как «import config»
├── init_db.py
├── sync_ad_users.py
├── add_users.py
├── requirements.txt
├── .env                          ← секреты и настройки (НЕ в git)
├── .venv/                        ← виртуальное окружение
├── deploy/                       ← образцы конфигов
└── app/                          ← ПАКЕТ приложения (читается как «import app»)
    ├── __init__.py               ← здесь create_app()
    ├── extensions.py
    ├── blueprints/  models/  services/  utils/  forms/
    ├── static/  templates/
```

**Почему так.** `run.py` делает `from app import create_app`, а
`app/__init__.py` делает `from config import Config`. Когда gunicorn стартует с
`WorkingDirectory=/opt/servicedesk`, текущая папка попадает в `sys.path`, и
одновременно доступны и пакет `app/`, и модуль `config.py`.

**Проверка раскладки (быстрый тест без Apache):**
```bash
cd /opt/servicedesk
sudo -u servicedesk .venv/bin/python -c "import run; print('OK', len(list(run.app.url_map.iter_rules())), 'маршрутов')"
```
- `OK 48 маршрутов` — раскладка верная.
- `ModuleNotFoundError: No module named 'app'` — пакет не в `app/` (свалили в кучу).
- `ModuleNotFoundError: No module named 'config'` — нет `config.py` в `/opt/servicedesk/`.

## 2. Главный урок: `curl -I` к gunicorn = `200`, а сайт = `404`. Почему это НЕ противоречие

Запрос проходит **три слоя**, и `curl -I http://127.0.0.1:8000/...` проверяет
только последний:

```
[браузер] --HTTPS--> [Apache :443]  --auth (Kerberos/Basic)-->  --proxy--> [gunicorn :8000] --> [Flask]
                         (слой 2)            (слой 3)                          (слой 1)
```

- `curl -I http://127.0.0.1:8000/...` бьёт **напрямую в gunicorn (слой 1)**,
  МИНУЯ Apache и авторизацию. `200 OK` тут означает только одно: **gunicorn жив
  и приложение отвечает**. Это НЕ доказывает, что работает HTTPS, проксирование
  и вход.
- `404` ты видел на слое 2–3: запрос шёл через Apache, и до приложения доходил
  либо неверный путь, либо приложение (после правок кода) **не было
  перезапущено** и обслуживало старую версию без нужного маршрута.

**Вывод-правило:** успех `curl` к `127.0.0.1:8000` и провал по HTTPS — это
нормальная, ожидаемая картина, которая прямо указывает: **проблема в слое
Apache/авторизация/проксирование, а не в gunicorn**. Не трать время на gunicorn,
если localhost отвечает — иди разбираться в Apache и заголовки.

> Полезно: добавляй в `curl` путь и смотри на `Server:` в ответе.
> `Server: gunicorn` + `404` = ответ сформировало приложение (значит запрос
> ДОШёл до gunicorn, но маршрут не найден). Обычная страница Apache = до
> приложения не дошло (vhost/ProxyPass).

## 2a. Частный случай: 404 на пути `/error/HTTP_UNAUTHORIZED.html.var`

Если диагностическая страница 404 показывает **путь `/error/HTTP_UNAUTHORIZED.html.var`**
(или `HTTP_FORBIDDEN`, `HTTP_BAD_GATEWAY` и т.п.) — это НЕ ошибка приложения, а
такая цепочка:

1. Apache по какой-то причине отдаёт ошибку (чаще всего **401** — доменная
   аутентификация не прошла);
2. у Apache включены многоязычные страницы ошибок
   (`ErrorDocument 401 /error/HTTP_UNAUTHORIZED.html.var`);
3. но `ProxyPass / http://127.0.0.1:8000/` проксирует в приложение **всё**,
   включая этот внутренний `/error/...` → приложение такого маршрута не имеет → 404.

**Что это означает по сути:** доменный вход (Kerberos/Basic) не сработал, а
страница ошибки 401 «провалилась» в приложение.

**Исправление (уже внесено):**
- `ProxyPass /error !` — страницы ошибок Apache больше не уходят в приложение;
- **Kerberos снят со всего сайта** и оставлен только на `/auth/sso` — поэтому
  обычная навигация и локальный вход больше не упираются в 401.

### Почему раньше даже ПОСЛЕ правильного локального логина была 404
Старый конфиг вешал Kerberos на весь сайт (`<Location />` + `Require valid-user`).
Локальный вход на `/auth/login` проходил, приложение делало редирект на `/`, но
Apache на `/` снова требовал домен → 401 → `/error/...` → 404. Apache не знает про
сессию Flask, поэтому «пускать после локального входа» он не умеет. Вывод:
**нельзя одновременно требовать домен на всём сайте и иметь локальный вход** —
домен оставляем только на точке `/auth/sso`.

## 2b. Новая схема входа (после исправления)

```
Пользователь → любая страница → приложение редиректит на /auth/login
   ├─ ЛОКАЛЬНАЯ учётка: ввести почту+пароль в форму (работает всегда, даже если домен лежит)
   └─ Кнопка «Войти по доменной учётной записи» → /auth/sso
         (на /auth/sso Apache делает Kerberos; если билета нет — спросит
          ДОМЕННЫЙ логин/пароль и проверит через KDC)
```

- Внутри сайта вход стережёт само приложение (`@login_required`).
- Доменная аутентификация — только на `/auth/sso` (кнопка на странице входа).
- Имя пользователя приходит заголовком `X-Remote-User`, приложение логинит
  пользователя из своей базы (выгруженной из AD).

Проверка, что Apache реально передаёт имя на `/auth/sso`: открой
`https://<сервер>/whoami` (этот путь Kerberos не защищает, но `/auth/sso` —
защищает; на `/whoami` поле `header_x_remote_user` будет пустым — это нормально).
Чтобы проверить ИМЕННО доменный handoff, зайди по кнопке «Войти по доменной…» и
посмотри в `journalctl -u servicedesk -e` строку с `remote_user=...` для запроса
`GET /auth/sso`.

## 3. Точная диагностика по шагам («лестница»)

Идём снизу вверх. На каждом шаге — одна команда и однозначный вывод.

**Шаг 1. Жив ли gunicorn и приложение?**
```bash
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8000/healthz
curl -s http://127.0.0.1:8000/healthz
```
- `{"status":"ok","db":"ok",...}` → приложение и БД живы → переходи к шагу 2.
- нет ответа / `Connection refused` → gunicorn не слушает: `systemctl status servicedesk`, `journalctl -u servicedesk -e`.
- `db: error` → проблема с БД/`.env` (`DATABASE_URL`).

**Шаг 2. Доходит ли через Apache по HTTPS?**
```bash
curl -kI https://srv01061.polyot.ru/healthz
```
- `200` → Apache проксирует корректно. Если при этом основной сайт даёт 404/401
  — дело в **авторизации/путях**, шаг 3.
- `404`/`500` со `Server:` Apache → vhost/`ProxyPass`: `apachectl -S`,
  `apachectl configtest`, смотри `/var/log/httpd2/sd_error.log`.

**Шаг 3. Передаёт ли Apache имя пользователя приложению? (ТО САМОЕ место)**
Открой в браузере (после доменной аутентификации) **`https://srv01061.polyot.ru/whoami`**
или:
```bash
curl -k --negotiate -u : https://srv01061.polyot.ru/whoami
```
Смотри поля:
- `header_x_remote_user`: должно содержать имя (например `POLYOT\ivanov`).
  - `null`/пусто → **Apache не пробрасывает имя**. Проверь в конфиге строку
    `RequestHeader set X-Remote-User "%{REMOTE_USER}s"`, модуль `headers`
    (`a2enmod headers`), keytab и что аутентификация вообще прошла.
- `env_remote_user`: при проксировании на gunicorn почти всегда `null` — **это
  нормально**, имя должно идти заголовком (в этом и была исходная ошибка: код
  читал `REMOTE_USER` из окружения, которого нет).
- `authenticated`: `true` → пользователь найден в БД и залогинен. `false` при
  заполненном `header_x_remote_user` → пользователя нет в базе (не
  синхронизирован из AD) или он `is_active=false`.

**Шаг 4. Логи (теперь информативные):**
```bash
journalctl -u servicedesk -e        # каждый запрос: метод, путь, код, remote_user, ip
tail -f /var/log/httpd2/sd_error.log    # ошибки ИМЕННО этого сайта
tail -f /var/log/httpd2/sd_access.log   # доступ ИМЕННО этого сайта
```
> Раньше ты смотрел общий `/var/log/httpd2/error_log` и `access_log` — там
> только служебные сообщения старта Apache. Журналы сайта — это `sd_*.log`
> (заданы директивами `ErrorLog`/`CustomLog` в vhost servicedesk).

## 4. Таблица: ошибка → причина → решение

| Симптом | Причина | Решение |
|---|---|---|
| `404 Not Found`, `Server: gunicorn` | запрос дошёл до приложения, но маршрута нет: старый код в gunicorn или неверный путь (`/login` вместо `/auth/login`) | `systemctl restart servicedesk` после правок; проверь путь; страница 404 теперь сама пишет путь и слой |
| 404 на пути `/error/HTTP_UNAUTHORIZED.html.var` | Apache отдал 401, его страница ошибки ушла в прокси | `ProxyPass /error !`; Kerberos снят со всего сайта, оставлен на `/auth/sso` (см. 2a/2b) |
| Локальный вход проходит, но затем 404 | Kerberos висел на всём сайте — редирект на `/` упирался в 401 | исправлено: домен только на `/auth/sso`, вход внутри сайта стережёт приложение |
| После Kerberos всё равно бросает на `/auth/login` | имя приходит заголовком `X-Remote-User`, код раньше читал `REMOTE_USER` из окружения | исправлено в `auth/routes.py` (читаем заголовок); проверь `/whoami` |
| Страница входа `/auth/login` недоступна, требует домен | в Apache исключения были на `/login`, а маршрут — `/auth/login` | исправлено: `<LocationMatch "^/auth/(login|logout)">` |
| `journalctl -u servicedesk` → «no entries» | gunicorn не писал в stdout | в `servicedesk.service` добавлены `--access-logfile - --error-logfile - --capture-output` |
| `curl` localhost `200`, а HTTPS `404` | проверяешь не тот слой; Apache/прокси/код не перезапущен | см. раздел 2 и «лестницу» раздела 3 |
| `No module named 'app'` / `'config'` | неверная раскладка файлов | раздел 1 (пакет в `app/`, скрипты в корне) |
| `502 Bad Gateway` в браузере | gunicorn не запущен или слушает не тот адрес | `systemctl status servicedesk`; `ProxyPass` → `http://127.0.0.1:8000/` |
| `500` на странице | ошибка в коде/шаблоне/БД | страница 500 + `journalctl -u servicedesk -e` (полный стек) |
| `sqlalchemy ... could not connect` / `db: error` в `/healthz` | неверный `DATABASE_URL`, БД не поднята, нет прав | проверь `.env`, `systemctl status postgresql`, доступы пользователя БД |
| `WWW-Authenticate: Negotiate` и браузер не входит сам | нет билета/SPN/keytab или вход не с доменной машины | проверь keytab (`klist -k`), SPN `HTTP/srv01061.polyot.ru`; запасной путь — `/auth/login` локальной учёткой |
| Чат/уведомления «разъезжаются» | несколько воркеров gunicorn | оставить `-w 1` (состояние Socket.IO — в памяти процесса) |

## 5. Обязательные переменные `.env`

```ini
# Безопасность и БД
SECRET_KEY=<длинная случайная строка>
DATABASE_URL=postgresql+psycopg2://servicedesk:<пароль>@127.0.0.1:5432/servicedesk

# Kerberos SSO: true на сервере в домене, false на машине без домена
KERBEROS_SSO_ENABLED=true

# Active Directory (узнать у админа; в КОД пароли НЕ писать!)
AD_LDAP_SERVER=ldaps://dc1.polyot.ru,ldaps://dc2.polyot.ru
AD_USE_SSL=true
AD_DOMAIN=POLYOT
AD_BASE_DN=OU=Person,OU=Users,OU=Local,DC=Polyot,DC=ru
AD_BIND_USER=POLYOT\<сервисная_учётка>
AD_BIND_PASSWORD=<пароль_сервисной_учётки>
AD_AUTH_METHOD=SIMPLE
AD_DEFAULT_ROLE=user

# Запасные локальные администраторы (создаются add_users.py)
BOOTSTRAP_ADMIN1_EMAIL=admin@polyot.ru
BOOTSTRAP_ADMIN1_PASSWORD=<пароль>
```

> ⚠️ Ранее `AD_BIND_PASSWORD` был зашит в `config.py` и попал в git — **этот
> пароль скомпрометирован, смените его в домене**. Теперь значения берутся только
> из `.env`, а `.env` исключён через `.gitignore`.

## 6. Чек-лист запуска (по порядку)

```bash
# 1. Раскладка верна и код импортируется
cd /opt/servicedesk && sudo -u servicedesk .venv/bin/python -c "import run; print(len(list(run.app.url_map.iter_rules())))"

# 2. Зависимости и БД
sudo -u servicedesk .venv/bin/pip install -r requirements.txt
sudo -u servicedesk .venv/bin/python init_db.py     # создать/применить схему
sudo -u servicedesk .venv/bin/python add_users.py   # запасные админы

# 3. Служба приложения
systemctl daemon-reload && systemctl restart servicedesk
curl -s http://127.0.0.1:8000/healthz                # ждём status: ok

# 4. Apache
apachectl configtest && systemctl reload httpd2
curl -kI https://srv01061.polyot.ru/healthz          # ждём 200

# 5. Проброс имени (после доменного входа)
#    открыть https://srv01061.polyot.ru/whoami  -> header_x_remote_user заполнен, authenticated: true
```

После ЛЮБОГО изменения кода — `systemctl restart servicedesk` (gunicorn не
перечитывает код сам). После изменения конфига Apache —
`apachectl configtest && systemctl reload httpd2`.
