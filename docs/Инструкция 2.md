# ServiceDesk на Ubuntu 24.04 — подробная инструкция «для новичка»

Эта инструкция — расширенная версия. Она объясняет **каждую команду** и
отдельно разбирает сложные. Если вы впервые работаете с Linux — начните с
**Части 1 (шпаргалка по командам)**, тогда всё остальное станет понятным.

Содержание:
1. Шпаргалка по командам Linux (что это и зачем)
2. Как редактировать файлы (редактор nano)
3. Как скопировать файлы с Windows на виртуалку
4. Установка и настройка PostgreSQL
5. Полное развёртывание шаг за шагом (с пояснением каждой команды)
6. Если что-то пошло не так

---

## Часть 1. Шпаргалка по командам Linux

### 1.1. Как вообще читать команды

Когда вы видите в терминале строку вида:
```
admin@servicedesk:~$
```
- `admin` — ваш пользователь, `servicedesk` — имя машины, `~` — где вы сейчас
  находитесь (тут — домашняя папка), `$` — приглашение «вводите команду».
- Если вместо `$` стоит `#` — вы работаете от имени администратора (root).

**Путь** — это «адрес» файла или папки:
- `/` — самый верх (корень всей системы);
- `~` — ваша домашняя папка (то же, что `/home/admin`);
- `.` — «текущая папка», `..` — «папка на уровень выше».

### 1.2. Передвижение и просмотр (безопасные команды, ничего не ломают)

| Команда | Что делает | Пример |
|--------|------------|--------|
| `pwd` | показать, в какой папке я сейчас | `pwd` |
| `ls` | показать список файлов в папке | `ls` |
| `ls -la` | то же, но подробно (размер, права, скрытые файлы) | `ls -la` |
| `cd папка` | зайти в папку | `cd /opt/servicedesk` |
| `cd ..` | подняться на уровень выше | `cd ..` |
| `cd ~` | вернуться в домашнюю папку | `cd ~` |
| `cat файл` | показать содержимое файла целиком | `cat .env` |
| `less файл` | показать файл с прокруткой (выход — клавиша `q`) | `less requirements.txt` |
| `head файл` | показать первые 10 строк файла | `head run.py` |
| `tail файл` | показать последние 10 строк | `tail logs/ad_sync.log` |
| `tail -f файл` | показывать файл и обновлять «вживую» (выход — `Ctrl+C`) | `tail -f logs/ad_sync.log` |

### 1.3. Создание, копирование, перемещение, удаление

| Команда | Что делает | Пример |
|--------|------------|--------|
| `mkdir папка` | создать папку | `mkdir logs` |
| `cp откуда куда` | **скопировать** файл (оригинал остаётся) | `cp .env.example .env` |
| `cp -r откуда куда` | скопировать папку со всем содержимым (`-r` = рекурсивно) | `cp -r sd_main /opt/servicedesk` |
| `mv откуда куда` | **переместить** или переименовать файл | `mv old.txt new.txt` |
| `rm файл` | **удалить** файл (без корзины! насовсем) | `rm test.txt` |
| `rm -r папка` | удалить папку со всем содержимым | `rm -r старая_папка` |

> ⚠️ `rm` удаляет **сразу и навсегда**, корзины нет. Удаляйте внимательно.
> Особенно НИКОГДА не пишите `rm -r /` или `rm -rf /*` — это сотрёт систему.

### 1.4. Права администратора: `sudo`

В Linux обычному пользователю нельзя менять системные файлы. Чтобы выполнить
команду «от имени администратора», перед ней пишут **`sudo`** (система спросит
ваш пароль).

- `sudo команда` — выполнить команду как администратор (root).
- `sudo -u имя команда` — выполнить команду от имени **другого** пользователя.
  Например `sudo -u servicedesk nano .env` — отредактировать файл от лица
  пользователя `servicedesk` (чтобы файл остался «его»).

### 1.5. Права на файлы (коротко): `chown` и `chmod`

У каждого файла есть **владелец** и **права доступа**.
- `sudo chown servicedesk:servicedesk файл` — сделать пользователя
  `servicedesk` владельцем файла (`chown` = change owner).
- `sudo chmod 640 файл` — задать права (кому читать/писать). Цифры — это уровни
  доступа; для наших задач конкретные значения даны ниже, заучивать не нужно.

### 1.6. Управление службами (сервисами): `systemctl`

Наше приложение работает как «служба», которой управляет `systemctl`:

| Команда | Что делает |
|--------|------------|
| `sudo systemctl start servicedesk` | запустить службу |
| `sudo systemctl stop servicedesk` | остановить |
| `sudo systemctl restart servicedesk` | перезапустить (после изменений) |
| `sudo systemctl enable servicedesk` | включить автозапуск при загрузке |
| `sudo systemctl enable --now servicedesk` | включить автозапуск И сразу запустить |
| `sudo systemctl status servicedesk` | посмотреть, работает ли (ищите `active (running)`) |
| `sudo journalctl -u servicedesk -f` | смотреть логи службы «вживую» (выход `Ctrl+C`) |

### 1.7. Установка программ: `apt`

| Команда | Что делает |
|--------|------------|
| `sudo apt-get update` | обновить список доступных программ (делать перед установкой) |
| `sudo apt-get install -y имя` | установить программу (`-y` = не переспрашивать) |

### 1.8. Мелочи, которые сильно экономят время

- **Tab** — автодополнение. Начните печатать имя файла и нажмите Tab — система
  допишет его сама.
- **Стрелка вверх (↑)** — повторить предыдущую команду (можно листать историю).
- **Ctrl + C** — прервать «зависшую» или бесконечную команду (например `tail -f`).
- **Вставить в терминал** скопированный текст: в окне Ubuntu/SSH — обычно
  **правый клик** или **Ctrl+Shift+V** (НЕ Ctrl+V).

### 1.9. Разбор «страшной» команды (ваш пример)

```
sudo cp /opt/servicedesk/deploy/servicedesk.service /etc/systemd/system/servicedesk.service
```
Читаем слева направо:
- `sudo` — выполнить от имени администратора (нужно, потому что папка `/etc` —
  системная, обычному пользователю туда писать нельзя);
- `cp` — команда «копировать»;
- `/opt/servicedesk/deploy/servicedesk.service` — **ОТКУДА** копируем
  (это файл-заготовка, который лежит в нашем проекте, в папке `deploy`);
- `/etc/systemd/system/servicedesk.service` — **КУДА** копируем
  (в системную папку, где `systemd` ищет описания служб).

Смысл одной фразой: **«скопируй наш файл-описание службы в системную папку,
чтобы Linux узнал про службу ServiceDesk и смог ею управлять»**.

## Часть 2. Как редактировать файлы (редактор nano)

`nano` — простой текстовый редактор прямо в терминале.

**Открыть файл:**
```
nano имя_файла
```
Если файл системный или чужой — добавьте `sudo`: `sudo nano имя_файла`.

**Внутри nano:**
- двигаться — стрелками; печатать — просто набирать текст;
- **сохранить** — `Ctrl + O`, затем нажать `Enter` (подтвердить имя файла);
- **выйти** — `Ctrl + X`;
- удалить целую строку — `Ctrl + K`;
- найти текст — `Ctrl + W`, ввести слово, `Enter`.

> Подсказки по горячим клавишам всегда видны внизу экрана nano. Значок `^`
> означает клавишу `Ctrl`. Например `^O` = `Ctrl+O`.

**Важно про права при редактировании `.env`.** Наш файл `.env` принадлежит
пользователю `servicedesk`. Самый надёжный способ его отредактировать:
```
sudo nano /opt/servicedesk/.env
# отредактировали, сохранили (Ctrl+O, Enter), вышли (Ctrl+X)
sudo chown servicedesk:servicedesk /opt/servicedesk/.env
```
Вторая строка возвращает файл во владение `servicedesk` (на случай, если
редактирование от root что-то поменяло). Если эту строку забыть — служба может
не прочитать файл.

## Часть 3. Как скопировать файлы с Windows на виртуалку

Самый удобный способ — программа **WinSCP** (графическая, «перетащил мышкой»).

### 3.1. Подготовка
На виртуалке при установке Ubuntu должен быть включён **OpenSSH** (мы включали
его галочкой). Проверьте, что он работает:
```
sudo systemctl status ssh
```
Должно быть `active (running)`. Если нет — установите:
```
sudo apt-get update && sudo apt-get install -y openssh-server
```
Узнайте IP виртуалки (понадобится для подключения):
```
ip a
```
Ищите строку `inet 192.168.x.x/24` — это и есть адрес.

### 3.2. Подключение через WinSCP (на Windows)
1. Скачайте и установите **WinSCP** (winscp.net).
2. Запустите, в окне входа укажите:
   - **Имя хоста:** IP виртуалки (например `192.168.1.50`);
   - **Порт:** `22`;
   - **Пользователь / Пароль:** ваши от Ubuntu (которые задавали при установке);
   - Протокол: **SFTP**.
3. Нажмите **«Войти»**. Слева — ваш компьютер, справа — виртуалка.
4. Справа вы окажетесь в домашней папке (`/home/admin`). **Перетащите** мышкой
   папку проекта `sd_main` слева направо. Файлы скопируются в `/home/admin/sd_main`.

### 3.3. Перенести проект в рабочую папку `/opt/servicedesk`
Файлы пока лежат в домашней папке. Переложим их в системную папку проекта
(в терминале виртуалки):
```
sudo mkdir -p /opt/servicedesk
sudo cp -r ~/sd_main/* /opt/servicedesk/
```
- `mkdir -p /opt/servicedesk` — создать папку проекта (`-p` = не ругаться, если
  уже есть);
- `cp -r ~/sd_main/* /opt/servicedesk/` — скопировать **всё содержимое** папки
  `sd_main` из дома (`~`) в `/opt/servicedesk` (`*` = «все файлы внутри»).

> Алтернатива WinSCP — команда `scp` прямо из PowerShell на Windows:
> ```
> scp -r C:\путь\к\sd_main admin@192.168.1.50:/home/admin/
> ```

## Часть 4. Установка и настройка PostgreSQL

У вас и дома, и на работе PostgreSQL — настроим его.

### 4.1. Установить PostgreSQL
```
sudo apt-get update
sudo apt-get install -y postgresql postgresql-contrib
```
После установки база уже работает как служба. Проверка:
```
sudo systemctl status postgresql
```

### 4.2. Создать базу данных и пользователя для приложения
PostgreSQL создаёт системного пользователя `postgres`. От его имени зайдём в
консоль базы данных:
```
sudo -u postgres psql
```
Приглашение сменится на `postgres=#`. Введите **по одной** эти строки
(пароль придумайте свой, лучше из латинских букв и цифр, без спецсимволов —
так проще, не нужно ничего экранировать):
```sql
CREATE USER servicedesk WITH PASSWORD 'MyPass123';
CREATE DATABASE servicedesk OWNER servicedesk;
GRANT ALL PRIVILEGES ON DATABASE servicedesk TO servicedesk;
\q
```
- первая строка — создаёт пользователя БД `servicedesk` с паролем;
- вторая — создаёт базу `servicedesk`, владелец — наш пользователь;
- третья — выдаёт ему полные права на эту базу;
- `\q` — выйти из консоли psql.

### 4.3. Прописать подключение в `.env`
Строка подключения (DATABASE_URL) для нашего примера будет такой:
```
DATABASE_URL=postgresql+psycopg2://servicedesk:MyPass123@localhost:5432/servicedesk
```
Расшифровка частей:
- `postgresql+psycopg2` — какая СУБД и какой драйвер;
- `servicedesk:MyPass123` — логин и пароль из шага 4.2;
- `localhost:5432` — адрес и порт БД (PostgreSQL по умолчанию на 5432, локально);
- `/servicedesk` — имя базы.

> Если в пароле есть спецсимволы (`@`, `:`, `/`), их пришлось бы «экранировать».
> Чтобы не усложнять — используйте простой пароль из букв и цифр.

### 4.4. Драйвер Python для PostgreSQL
В `requirements.txt` уже добавлен `psycopg2-binary` — это и есть драйвер,
через который Python общается с PostgreSQL. Отдельно ставить не нужно, он
поставится вместе с остальными зависимостями (Часть 5).

### 4.5. Для офлайна (работа)
Не забудьте положить в комплект на флешку:
- системные пакеты PostgreSQL:
  ```
  sudo apt-get install --download-only --reinstall -y postgresql postgresql-contrib
  ```
  (потом `sudo cp /var/cache/apt/archives/*.deb ~/offline-bundle/deb/`);
- драйвер `psycopg2-binary` уже попадёт в `wheels` при `pip download -r requirements.txt`.

## Часть 5. Полное развёртывание шаг за шагом

Здесь — весь процесс подряд, каждая команда с пояснением. Дома (с интернетом)
выполняйте как есть; на работе (офлайн) отличаются только шаги установки
пакетов — см. пометки «офлайн».

### Шаг 1. Обновить систему и поставить базовые программы
```
sudo apt-get update
sudo apt-get install -y python3 python3-venv python3-pip apache2 postgresql postgresql-contrib
```
- `python3`, `python3-venv`, `python3-pip` — Python и инструменты к нему;
- `apache2` — веб-сервер (обратный прокси перед приложением);
- `postgresql` — база данных.

> **Офлайн:** вместо этого ставим из папки с флешки:
> `sudo dpkg -i ~/offline-bundle/deb/*.deb`

### Шаг 2. Создать отдельного пользователя для приложения
```
sudo useradd --system --shell /usr/sbin/nologin servicedesk
```
- создаёт служебного пользователя `servicedesk` без возможности логиниться
  (так безопаснее — приложение работает не под вашим личным аккаунтом).

### Шаг 3. Положить код в `/opt/servicedesk` и отдать его пользователю
(Файлы вы уже скопировали в Части 3.)
```
sudo chown -R servicedesk:servicedesk /opt/servicedesk
```
- `chown -R` — сделать `servicedesk` владельцем папки и **всего внутри**
  (`-R` = рекурсивно).

### Шаг 4. Создать виртуальное окружение и поставить зависимости
```
cd /opt/servicedesk
sudo -u servicedesk python3 -m venv .venv
sudo -u servicedesk .venv/bin/pip install --upgrade pip
sudo -u servicedesk .venv/bin/pip install -r requirements.txt
```
- `cd /opt/servicedesk` — перейти в папку проекта;
- `python3 -m venv .venv` — создать «изолированный ящик» для библиотек проекта
  (папка `.venv`), чтобы не мешать системному Python;
- `pip install -r requirements.txt` — поставить все библиотеки из списка.

> **Офлайн:** последняя команда другая (берём из папки на флешке):
> ```
> sudo -u servicedesk .venv/bin/pip install --no-index --find-links=$HOME/offline-bundle/wheels -r requirements.txt
> ```
> `--no-index` = не лезть в интернет, `--find-links` = брать файлы из папки.

### Шаг 5. Настроить файл `.env`
Создаём свой `.env` из шаблона и открываем для редактирования:
```
sudo -u servicedesk cp /opt/servicedesk/.env.example /opt/servicedesk/.env
sudo nano /opt/servicedesk/.env
```
Заполните как минимум:
- `SERCET_KEY=` — длинная случайная строка. Сгенерировать её можно так
  (выполните отдельно и вставьте результат):
  ```
  /opt/servicedesk/.venv/bin/python -c "import secrets; print(secrets.token_hex(32))"
  ```
- `DATABASE_URL=postgresql+psycopg2://servicedesk:MyPass123@localhost:5432/servicedesk`
  (из Части 4.3);
- дома: `KERBEROS_SSO_ENABLED=false`;
- `BOOTSTRAP_ADMIN_EMAIL=` и `BOOTSTRAP_ADMIN_PASSWORD=` — чтобы создать
  локального админа для первого входа.

Сохраните (`Ctrl+O`, `Enter`, `Ctrl+X`) и верните владельца:
```
sudo chown servicedesk:servicedesk /opt/servicedesk/.env
```

### Шаг 6. Создать таблицы в базе
```
sudo -u servicedesk .venv/bin/python init_db.py
```
- скрипт создаёт все нужные таблицы в PostgreSQL и (если задали логин/пароль
  в `.env`) запасного локального администратора.

### Шаг 7. Быстрая проверка, что приложение поднимается
```
sudo -u servicedesk .venv/bin/gunicorn -k gthread -w 1 --threads 8 -b 0.0.0.0:8000 run:app
```
Откройте в браузере: `http://<IP-виртуалки>:8000` — должна появиться страница
входа. Войдите локальным админом. Проверили — остановите командой `Ctrl + C`.

### Шаг 8. Сделать автозапуск через systemd
```
sudo cp /opt/servicedesk/deploy/servicedesk.service /etc/systemd/system/servicedesk.service
sudo systemctl daemon-reload
sudo systemctl enable --now servicedesk
sudo systemctl status servicedesk
```
- первая строка (разбор — в п.1.9) кладёт описание службы в системную папку;
- `daemon-reload` — сказать systemd «перечитай описания служб»;
- `enable --now` — включить автозапуск и сразу запустить;
- `status` — убедиться, что `active (running)`.

> Если в статусе ошибка — смотрите подробности: `sudo journalctl -u servicedesk -e`

### Шаг 9. Поставить Apache «перед» приложением (порт 80)
```
sudo a2enmod proxy proxy_http headers
sudo cp /opt/servicedesk/deploy/apache-servicedesk.conf /etc/apache2/sites-available/servicedesk.conf
sudo a2dissite 000-default
sudo a2ensite servicedesk
sudo systemctl restart apache2
```
- `a2enmod …` — включить модули Apache, нужные для проксирования;
- `cp …` — положить нашу конфигурацию сайта;
- `a2dissite 000-default` — выключить стандартную заглушку Apache;
- `a2ensite servicedesk` — включить наш сайт;
- `restart apache2` — перезапустить Apache, чтобы применить.

Теперь сервис открывается по адресу виртуалки на обычном порту 80
(`http://<IP-виртуалки>`).

### Шаг 10 (только работа). Домен и Kerberos
Это делается на работе, в домене. Подробно — в `docs/LINUX_DEPLOY_SETUP.md`
(Часть D) и `docs/AD_KERBEROS_IIS_SETUP.md` (чек-лист данных). Кратко:
ввести сервер в домен (`realm join`), сделать keytab, в `.env` поставить
`KERBEROS_SSO_ENABLED=true`, в `apache-servicedesk.conf` раскомментировать блок
Kerberos, запустить `sync_ad_users.py` (первичная выгрузка) и завести ночной cron.

## Часть 6. Если что-то пошло не так

| Проблема | Что проверить / что сделать |
|----------|------------------------------|
| Не получается скопировать файлы с Windows | Проверьте, что на ВМ работает SSH (`sudo systemctl status ssh`) и что вы вводите правильный IP/логин/пароль в WinSCP |
| `Permission denied` при правке файла | Файл системный или чужой — добавьте `sudo` перед командой (например `sudo nano …`) |
| Изменил `.env`, но служба «не видит» новых значений | После правки выполните `sudo chown servicedesk:servicedesk /opt/servicedesk/.env` и `sudo systemctl restart servicedesk` |
| `init_db.py`/приложение не подключается к базе | Проверьте `DATABASE_URL` в `.env` (логин/пароль/имя базы), что PostgreSQL запущен (`sudo systemctl status postgresql`) и что пользователь/база созданы (Часть 4.2) |
| Сайт не открывается на :8000 | Запущена ли служба: `sudo systemctl status servicedesk`; логи: `sudo journalctl -u servicedesk -e`; firewall: `sudo ufw allow 8000/tcp` |
| `502 Bad Gateway` от Apache (порт 80) | Не запущено приложение (gunicorn). Сначала почините службу `servicedesk`, потом обновите страницу |
| Случайно удалил файл командой `rm` | Восстановить нельзя (корзины нет) — скопируйте файл заново из проекта/с флешки |
| Забыл пароль локального админа | Запустите `init_db.py` снова с новыми `BOOTSTRAP_ADMIN_*` в `.env` (существующий с таким email не пересоздаётся — задайте другой email или удалите старого через psql) |

## Очень короткая шпаргалка (когда уже всё понятно)

```
# базовое
sudo apt-get update
sudo apt-get install -y python3 python3-venv python3-pip apache2 postgresql postgresql-contrib

# БД
sudo -u postgres psql        # внутри: CREATE USER ...; CREATE DATABASE ...; \q

# приложение
sudo useradd --system --shell /usr/sbin/nologin servicedesk
sudo cp -r ~/sd_main/* /opt/servicedesk/
sudo chown -R servicedesk:servicedesk /opt/servicedesk
cd /opt/servicedesk
sudo -u servicedesk python3 -m venv .venv
sudo -u servicedesk .venv/bin/pip install -r requirements.txt
sudo -u servicedesk cp .env.example .env && sudo nano .env   # заполнить, потом chown
sudo -u servicedesk .venv/bin/python init_db.py

# служба + Apache
sudo cp deploy/servicedesk.service /etc/systemd/system/
sudo systemctl daemon-reload && sudo systemctl enable --now servicedesk
sudo a2enmod proxy proxy_http headers
sudo cp deploy/apache-servicedesk.conf /etc/apache2/sites-available/servicedesk.conf
sudo a2dissite 000-default && sudo a2ensite servicedesk && sudo systemctl restart apache2
```
