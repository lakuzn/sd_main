# ServiceDesk — единое руководство по развёртыванию (для новичка)

Структура:
- **Часть 1.** Развёртывание на ВМ дома (с интернетом).
- **Часть 2.** Что скачать дома и как (офлайн-комплект для работы).
- **Часть 3.** Развёртывание на ВМ на работе (без интернета, в домене).
- **Часть 4.** Шпаргалка по Linux.

Шаги идут строго по порядку. Не пропускайте их — порядок выбран так, чтобы не
возникало ошибок с правами доступа и поиском файлов.

> ⚠️ Слова `CHANGE-ME`, `192.168.1.50`, `company.local`, `MyPass123` — это
> **примеры/заглушки**. Подставляйте свои значения (доменные — у сисадмина).

> 💡 Если вы впервые в Linux — сначала прочитайте **Часть 4 (шпаргалка)**,
> там объяснены все команды, которые встречаются ниже.

# ЧАСТЬ 1. Развёртывание на ВМ ДОМА (с интернетом)

Дома есть интернет, поэтому всё ставим обычным способом. Цель — увидеть, как
приложение работает, и заодно понять каждый шаг (на работе будет то же самое).

## 1.1. Создать виртуальную машину (Hyper-V) и поставить Ubuntu

1. Включите Hyper-V: «Панель управления» → «Программы» → «Включение/отключение
   компонентов Windows» → галочка **Hyper-V** → перезагрузка.
2. Скачайте образ **Ubuntu Server 24.04 LTS** с https://ubuntu.com/download/server
   (файл `ubuntu-24.04.x-live-server-amd64.iso`). Сохраните его — он же поедет
   на флешку для работы.
3. В Hyper-V создайте сеть: «Диспетчер виртуальных коммутаторов» → «Создать» →
   тип **«Внешний» (External)** → ваша сетевая карта → имя `ExternalSwitch`.
4. «Создать → Виртуальная машина»: поколение **2**, память **2048–4096 МБ**,
   сеть `ExternalSwitch`, диск **20–30 ГБ**, установка с образа (укажите `.iso`).
5. После создания: ПКМ по ВМ → «Параметры» → «Безопасность» → **снимите галочку
   «Включить безопасную загрузку»** (иначе Ubuntu может не загрузиться).
6. Запустите ВМ, пройдите установщик Ubuntu (язык, сеть — автоматически).
   - придумайте пользователя (например `admin`) и пароль — запомните их;
   - **обязательно** отметьте **«Install OpenSSH server»** (нужно для копирования
     файлов с Windows).
7. После установки войдите под `admin`. Узнайте IP машины:
   ```
   ip a
   ```
   Запишите адрес из строки `inet 192.168.x.x/24`.

## 1.2. Обновить систему и поставить программы

В терминале виртуалки:
```
sudo apt-get update
sudo apt-get install -y python3 python3-venv python3-pip apache2 postgresql postgresql-contrib
```
- `python3`, `python3-venv`, `python3-pip` — Python и инструменты к нему;
- `apache2` — веб-сервер (будет «прокси» перед приложением);
- `postgresql` — база данных.

## 1.3. Настроить PostgreSQL (создать базу и пользователя)

Войдите в консоль базы от системного пользователя `postgres`:
```
sudo -u postgres psql
```
Приглашение станет `postgres=#`. Введите построчно (пароль придумайте простой —
латиница и цифры, без `@ : /`, чтобы потом не экранировать):
```sql
CREATE USER servicedesk WITH PASSWORD 'MyPass123';
CREATE DATABASE servicedesk OWNER servicedesk;
GRANT ALL PRIVILEGES ON DATABASE servicedesk TO servicedesk;
\q
```
Запомните строку подключения — пригодится в `.env`:
```
postgresql+psycopg2://servicedesk:MyPass123@localhost:5432/servicedesk
```

## 1.4. Создать пользователя приложения (ДО копирования кода)

```
sudo useradd --system --shell /usr/sbin/nologin servicedesk
```
Это служебный пользователь, от имени которого будет работать приложение
(безопаснее, чем под вашим личным аккаунтом).

## 1.5. Скопировать код проекта с Windows на виртуалку

Файлы передаём по SSH программой **WinSCP** (на Windows):
1. Скачайте и установите WinSCP (winscp.net).
2. Подключение: **Имя хоста** = IP виртуалки, **Порт** = `22`, **Пользователь/
   Пароль** = ваши от Ubuntu, протокол **SFTP** → «Войти».
3. Справа — виртуалка (вы в `/home/admin`). Перетащите мышкой папку проекта
   `sd_main` слева направо. Она окажется в `/home/admin/sd_main`.

Теперь в терминале виртуалки переложите код в рабочую папку и **сразу** отдайте
её пользователю `servicedesk` (важно для прав):
```
sudo mkdir -p /opt/servicedesk
sudo cp -r ~/sd_main/* /opt/servicedesk/
sudo chown -R servicedesk:servicedesk /opt/servicedesk
```
> Порядок важен: сначала скопировали файлы, потом одной командой отдали всю
> папку владельцу `servicedesk`. Дальше всё, что создаёт приложение, тоже будет
> «его», и не возникнет ошибок доступа.

## 1.6. Создать окружение и поставить библиотеки

```
cd /opt/servicedesk
sudo -u servicedesk python3 -m venv .venv
sudo -u servicedesk .venv/bin/pip install --upgrade pip
sudo -u servicedesk .venv/bin/pip install -r requirements.txt
```
- `cd /opt/servicedesk` — перейти в папку проекта;
- `venv .venv` — создать «изолированный ящик» для библиотек проекта;
- `pip install -r requirements.txt` — поставить все библиотеки из списка
  (Flask, ldap3, gunicorn, psycopg2-binary и т.д.).

> Обратите внимание: команды Python/pip запускаем через **`sudo -u servicedesk`**,
> чтобы создаваемые файлы принадлежали `servicedesk`, а не root.

## 1.7. Заполнить файл `.env`

Создаём свой `.env` из шаблона (от имени servicedesk) и редактируем:
```
sudo -u servicedesk cp /opt/servicedesk/.env.example /opt/servicedesk/.env
sudo nano /opt/servicedesk/.env
```
Заполните как минимум:
- `SERCET_KEY=` — длинная случайная строка. Сгенерировать (выполните отдельно,
  результат вставьте в файл):
  ```
  /opt/servicedesk/.venv/bin/python -c "import secrets; print(secrets.token_hex(32))"
  ```
- `DATABASE_URL=postgresql+psycopg2://servicedesk:MyPass123@localhost:5432/servicedesk`
- `KERBEROS_SSO_ENABLED=false` (дома домена нет → вход по логину/паролю);
- `BOOTSTRAP_ADMIN_EMAIL=admin@local` и `BOOTSTRAP_ADMIN_PASSWORD=...` — для входа.

Сохраните (`Ctrl+O`, `Enter`, `Ctrl+X`) и **верните владельца** (обязательно!):
```
sudo chown servicedesk:servicedesk /opt/servicedesk/.env
```
> Если забыть про `chown` — служба не сможет прочитать файл. Это частая ошибка.

## 1.8. Создать таблицы в базе

```
sudo -u servicedesk .venv/bin/python init_db.py
```
Скрипт создаст таблицы в PostgreSQL и (раз задали `BOOTSTRAP_ADMIN_*`)
запасного локального администратора.

## 1.9. Быстрая проверка «вживую»

```
sudo -u servicedesk .venv/bin/gunicorn -k gthread -w 1 --threads 8 -b 0.0.0.0:8000 run:app
```
Откройте в браузере: `http://<IP-виртуалки>:8000` — должна появиться страница
входа, войдите локальным админом. Проверили — остановите клавишами `Ctrl + C`.

## 1.10. Автозапуск через systemd

```
sudo cp /opt/servicedesk/deploy/servicedesk.service /etc/systemd/system/servicedesk.service
sudo systemctl daemon-reload
sudo systemctl enable --now servicedesk
sudo systemctl status servicedesk
```
- 1-я строка кладёт описание службы в системную папку (`/etc/systemd/system`),
  откуда systemd читает службы;
- `daemon-reload` — перечитать описания;
- `enable --now` — включить автозапуск и сразу запустить;
- `status` — убедиться, что `active (running)` (выход из просмотра — `q`).

## 1.11. Apache «перед» приложением (порт 80)

```
sudo a2enmod proxy proxy_http headers
sudo cp /opt/servicedesk/deploy/apache-servicedesk.conf /etc/apache2/sites-available/servicedesk.conf
sudo a2dissite 000-default
sudo a2ensite servicedesk
sudo systemctl restart apache2
```
Теперь сервис открывается по `http://<IP-виртуалки>` (обычный порт 80).

**Дома готово.** Вы прошли весь путь: пакеты → база → код → окружение →
служба → Apache.

# ЧАСТЬ 2. Что скачать дома и как (комплект для работы)

На работе интернета нет, поэтому **всё** скачиваем дома и несём на флешке.
Делать это нужно на **такой же** Ubuntu (та же версия 24.04), что и на работе.

Создаём папку комплекта и подпапки:
```
mkdir -p ~/offline-bundle/deb ~/offline-bundle/wheels
```

В итоге у нас будет вот такая структура (объясняю каждую папку ниже):
```
offline-bundle/
├─ ubuntu-24.04.x-live-server-amd64.iso   ← образ ОС для установки на работе
├─ deb/        ← системные программы (.deb): Apache, PostgreSQL, домен, Kerberos
├─ wheels/     ← Python-библиотеки проекта (.whl)
└─ sd_main/    ← код самого приложения ServiceDesk
```

## 2.1. Системные пакеты (папка `deb/`)

Скачиваем (НЕ устанавливаем) все программы, которые понадобятся на сервере,
вместе с их зависимостями — включая всё для домена и Kerberos:
```
sudo apt-get update
sudo apt-get install --download-only --reinstall -y \
  python3 python3-venv python3-pip \
  apache2 libapache2-mod-auth-gssapi \
  postgresql postgresql-contrib \
  krb5-user sssd sssd-tools realmd adcli \
  samba-common-bin oddjob oddjob-mkhomedir packagekit libnss-sss libpam-sss
sudo cp /var/cache/apt/archives/*.deb ~/offline-bundle/deb/
sudo chown -R $USER:$USER ~/offline-bundle/deb
```
Что есть что:
- `python3*` — Python и инструменты;
- `apache2`, `libapache2-mod-auth-gssapi` — веб-сервер и **модуль Kerberos** для него;
- `postgresql`, `postgresql-contrib` — база данных;
- `krb5-user` — клиент Kerberos;
- `sssd`, `sssd-tools`, `realmd`, `adcli`, `samba-common-bin`, `libnss-sss`,
  `libpam-sss`, `oddjob*`, `packagekit` — всё для **ввода сервера в домен AD**.

Ключевые флаги:
- `--download-only` — только скачать, не ставить;
- `--reinstall` — скачать пакеты, даже если они уже стоят на этой ВМ (иначе на
  работе может не хватить «уже установленного дома» пакета).

> 🔎 Чтобы наверняка скачать ВСЁ: лучше делать это на **свежей** виртуалке,
> установленной с того же `.iso`. Тогда набор базовых пакетов совпадает с
> рабочим сервером, и ничего не «потеряется».

## 2.2. Python-библиотеки (папка `wheels/`)

Эти файлы платформонезависимы (чистый Python), их качаем из интернета:
```
cd /opt/servicedesk
sudo -u servicedesk .venv/bin/pip download -r requirements.txt -d ~/offline-bundle/wheels
```
В папку `wheels/` попадут все библиотеки из `requirements.txt` **и их
под-зависимости** (Werkzeug, SQLAlchemy, psycopg2-binary и т.д.) — pip скачает
их автоматически. Отдельно ничего перечислять не нужно.

## 2.3. Образ ОС и код проекта

Положите в `~/offline-bundle`:
- образ `ubuntu-24.04.x-live-server-amd64.iso` (тот же, что ставили дома);
- папку с кодом `sd_main` (целиком);
- при желании — эти инструкции (`docs/`).

```
cp ~/Downloads/ubuntu-24.04*-live-server-amd64.iso ~/offline-bundle/   # путь к образу — свой
cp -r ~/sd_main ~/offline-bundle/
```

## 2.4. Перенести комплект на флешку

С виртуалки на Windows проще всего — по SSH (WinSCP): подключитесь, перетащите
папку `offline-bundle` с виртуалки (справа) на компьютер (слева), а затем
скопируйте её на флешку. Всё, комплект готов.

# ЧАСТЬ 3. Развёртывание на ВМ на РАБОТЕ (офлайн, в домене)

Шаги почти те же, что дома, но программы ставим **из папки на флешке**, плюс
вводим сервер в домен и включаем Kerberos.

## 3.1. Поставить виртуалку и ОС
Как в 1.1, но образ `.iso` берёте с флешки. Версия Ubuntu — **та же**, что дома.
Сетевые настройки уточните у сисадмина (статический IP, маска, шлюз) и
**обязательно DNS — на контроллер домена** (иначе домен/Kerberos не заработают).
Также проверьте, что **время** на сервере совпадает с доменом (Kerberos
чувствителен ко времени): `timedatectl`.

## 3.2. Скопировать комплект с флешки на сервер
Скопируйте папку `offline-bundle` с флешки на сервер (по SSH с рабочего ПК).
Разложите код и создайте пользователя приложения:
```
sudo useradd --system --shell /usr/sbin/nologin servicedesk
sudo mkdir -p /opt/servicedesk
sudo cp -r ~/offline-bundle/sd_main/* /opt/servicedesk/
sudo chown -R servicedesk:servicedesk /opt/servicedesk
```

## 3.3. Установить системные пакеты ОФЛАЙН (из папки deb/)
```
cd ~/offline-bundle/deb
sudo dpkg -i *.deb
sudo apt-get install -f --no-download
```
- `dpkg -i *.deb` — установить все скачанные пакеты;
- если `dpkg` ругается на порядок зависимостей — **повторите** `sudo dpkg -i *.deb`
  ещё раз (со второго прохода обычно доустанавливается всё).

## 3.4. Настроить PostgreSQL
Как в 1.3 (создать пользователя и базу через `sudo -u postgres psql`).

## 3.5. Окружение и библиотеки ОФЛАЙН (из папки wheels/)
```
cd /opt/servicedesk
sudo -u servicedesk python3 -m venv .venv
sudo -u servicedesk .venv/bin/pip install --no-index --find-links=$HOME/offline-bundle/wheels -r requirements.txt
```
- `--no-index` — не выходить в интернет;
- `--find-links=...` — брать библиотеки из папки `wheels`.

## 3.6. Заполнить `.env` (боевой)
Как в 1.7, но:
- `KERBEROS_SSO_ENABLED=true`;
- заполнить все `AD_*` (адрес контроллера домена, Base DN, сервисная учётка и
  т.д. — чек-лист в `docs/AD_KERBEROS_IIS_SETUP.md`, раздел 1).
Не забудьте после правки: `sudo chown servicedesk:servicedesk /opt/servicedesk/.env`.

## 3.7. Ввести сервер в домен
```
realm discover company.local
sudo realm join --user=Администратор company.local
realm list
```
- `realm discover` — проверить, что домен виден (если нет — проблема в DNS, см. 3.1);
- `realm join` — ввести сервер в домен (спросит пароль учётки с правом ввода).

## 3.8. Keytab для веб-службы (Kerberos)
Чтобы Apache мог проверять Kerberos-билеты, нужен файл **keytab** для службы
`HTTP/<имя-сервера>` (например `HTTP/servicedesk.company.local`). Его делает/
выдаёт сисадмин. Положите файл и задайте права:
```
sudo chown root:www-data /etc/apache2/servicedesk.keytab
sudo chmod 640 /etc/apache2/servicedesk.keytab
```
> Точную процедуру и имя службы (SPN) согласуйте с сисадмином —
> **<<< УЗНАТЬ У СИСАДМИНА: keytab и SPN >>>**

## 3.9. Создать таблицы и выгрузить пользователей AD
```
sudo -u servicedesk .venv/bin/python init_db.py
sudo -u servicedesk .venv/bin/python sync_ad_users.py
```
- `init_db.py` — создать таблицы;
- `sync_ad_users.py` — **первичная выгрузка всех пользователей** из AD
  (лог пишется в `logs/ad_sync.log`).

## 3.10. Служба и Apache (с Kerberos)
```
sudo cp /opt/servicedesk/deploy/servicedesk.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now servicedesk

sudo a2enmod proxy proxy_http headers auth_gssapi
sudo cp /opt/servicedesk/deploy/apache-servicedesk.conf /etc/apache2/sites-available/servicedesk.conf
sudo nano /etc/apache2/sites-available/servicedesk.conf
#   -> впишите реальный ServerName
#   -> РАСКОММЕНТИРУЙТЕ блок <Location /> с Kerberos (AuthType GSSAPI ...)
sudo a2dissite 000-default
sudo a2ensite servicedesk
sudo systemctl restart apache2
```

## 3.11. Ночная синхронизация пользователей (cron)
```
sudo crontab -u servicedesk -e
```
Впишите строку (запуск каждую ночь в 03:00):
```
0 3 * * * cd /opt/servicedesk && /opt/servicedesk/.venv/bin/python sync_ad_users.py >> /opt/servicedesk/logs/cron.log 2>&1
```

## 3.12. Проверка входа по Kerberos
На **доменном** рабочем ПК откройте `http://servicedesk.company.local`. Если
сайт в зоне «Местная интрасеть» (обычно настроено групповой политикой), вход
произойдёт автоматически под вашей доменной учёткой, без ввода пароля.

# ЧАСТЬ 4. Шпаргалка по Linux

## 4.1. Как читать строку терминала
```
admin@servicedesk:~$
```
`admin` — пользователь, `servicedesk` — имя машины, `~` — текущая папка (тут
домашняя), `$` — приглашение вводить команду (`#` — вы под администратором).

**Пути:** `/` — корень системы; `~` — ваша домашняя папка; `.` — текущая папка;
`..` — папка уровнем выше.

## 4.2. Перемещение и просмотр (ничего не ломают)
| Команда | Что делает |
|--------|------------|
| `pwd` | показать, в какой папке я сейчас |
| `ls` | список файлов; `ls -la` — подробно, со скрытыми |
| `cd папка` | зайти в папку; `cd ..` — вверх; `cd ~` — домой |
| `cat файл` | показать файл целиком |
| `less файл` | показать с прокруткой (выход — `q`) |
| `tail -f файл` | показывать файл «вживую» (выход — `Ctrl+C`) |

## 4.3. Файлы и папки
| Команда | Что делает |
|--------|------------|
| `mkdir папка` | создать папку (`-p` — не ругаться, если есть) |
| `cp откуда куда` | копировать файл; `cp -r` — папку целиком |
| `mv откуда куда` | переместить/переименовать |
| `rm файл` | удалить файл (НАСОВСЕМ, корзины нет!) |
| `rm -r папка` | удалить папку со всем содержимым |

> ⚠️ `rm` удаляет безвозвратно. Никогда не пишите `rm -rf /` — это убьёт систему.

## 4.4. Права администратора
- `sudo команда` — выполнить как администратор (спросит ваш пароль);
- `sudo -u имя команда` — выполнить от имени другого пользователя
  (например `sudo -u servicedesk nano .env`).

## 4.5. Владелец и права файла
- `sudo chown servicedesk:servicedesk файл` — сделать `servicedesk` владельцем;
- `sudo chmod 640 файл` — задать права доступа (конкретные значения берите из
  инструкции, заучивать не нужно).

## 4.6. Службы (сервисы)
| Команда | Что делает |
|--------|------------|
| `sudo systemctl start/stop/restart имя` | запустить / остановить / перезапустить |
| `sudo systemctl enable --now имя` | включить автозапуск и сразу запустить |
| `sudo systemctl status имя` | проверить, работает ли (ищите `active (running)`) |
| `sudo journalctl -u имя -f` | смотреть логи «вживую» (выход — `Ctrl+C`) |

## 4.8. Редактор nano (правка файлов)
- открыть: `nano файл` (системный — `sudo nano файл`);
- двигаться — стрелками, печатать — обычным набором;
- **сохранить** — `Ctrl+O`, затем `Enter`;
- **выйти** — `Ctrl+X`;
- удалить строку — `Ctrl+K`; найти текст — `Ctrl+W`.
- Внизу экрана всегда видны подсказки; `^` означает `Ctrl`.

## 4.9. Мелочи, экономящие время
- **Tab** — автодополнение имени файла/папки;
- **↑ (стрелка вверх)** — повторить прошлую команду;
- **Ctrl+C** — прервать зависшую/бесконечную команду;
- вставка в терминал — **правый клик** или **Ctrl+Shift+V**.

## 4.10. Разбор сложной команды (на примере)
```
sudo cp /opt/servicedesk/deploy/servicedesk.service /etc/systemd/system/servicedesk.service
```
- `sudo` — от имени администратора (папка `/etc` системная, обычному
  пользователю туда нельзя);
- `cp` — копировать;
- `/opt/.../deploy/servicedesk.service` — **ОТКУДА** (файл-заготовка в проекте);
- `/etc/systemd/system/servicedesk.service` — **КУДА** (системная папка служб).

Смысл: «скопируй описание нашей службы в системную папку, чтобы Linux узнал про
службу ServiceDesk».

## 4.11. Если что-то пошло не так
| Проблема | Что сделать |
|----------|-------------|
| `Permission denied` | добавьте `sudo` перед командой |
| Изменил `.env`, служба не видит | `sudo chown servicedesk:servicedesk .env`, затем `sudo systemctl restart servicedesk` |
| Сайт не открывается на :8000 | `sudo systemctl status servicedesk`; логи: `sudo journalctl -u servicedesk -e` |
| `502 Bad Gateway` (порт 80) | приложение не запущено — почините службу `servicedesk` |
| Не подключается к базе | проверьте `DATABASE_URL` и что PostgreSQL запущен (`systemctl status postgresql`) |
| `realm discover` не находит домен | DNS не указывает на контроллер домена; проверьте время (`timedatectl`) |
