# Развёртывание ServiceDesk на Linux (VM): сначала дома, потом в домене

Эта инструкция написана «с нуля», для человека, который впервые ставит
Linux-сервер. Сценарий ровно ваш:

1. **Дома** поднимаем виртуальную машину с Linux и запускаем ServiceDesk
   локально (без домена) — чтобы посмотреть, как всё работает.
2. **Дома же** скачиваем ВСЕ файлы, которые понадобятся на работе (там нет
   интернета), и складываем их на флешку.
3. **На работе** ставим такую же виртуалку, копируем туда файлы с флешки,
   ставим всё офлайн, вводим сервер в домен и включаем вход по Kerberos.

> ⚠️ Слова `CHANGE-ME`, `servicedesk.local`, `company.local`, `COMPANY` и т.п. —
> это **заглушки**. Реальные значения (особенно доменные) узнавайте у сисадмина.

---

## 0. Три решения, которые приняты заранее (и почему)

**(а) Гипервизор: Hyper-V — да, подойдёт.** Вы хотели Hyper-V — он встроен в
Windows Pro/Enterprise и работает быстро. Единственные нюансы для новичка:
сеть нужно настроить через «External switch», а файлы в Linux-гостя удобнее
всего передавать по SSH (покажу как). Если захотите чего-то ещё проще в плане
«перетащить папку в виртуалку» и USB — **VirtualBox** дружелюбнее (общие папки
и проброс USB «из коробки»), но он не обязателен. **Вывод: оставайтесь на
Hyper-V**, в инструкции я учёл его нюансы.

**(б) Дистрибутив: Ubuntu Server 24.04 LTS** (64-бит). Причины: очень много
документации, простая установка, отличная интеграция с Active Directory
(`realm join`), и нужный для Kerberos модуль Apache есть в готовом виде.
*Подойдёт и 22.04 LTS — тогда просто другая версия Python (3.10 вместо 3.12),
на работу логики это не влияет.*

**(в) Python: системный, из дистрибутива (3.12), а НЕ 3.14.** Именно из-за
3.14 у вас не собирался пакет под IIS. На сервере берём Python, который идёт
вместе с Ubuntu — под него все наши библиотеки ставятся готовыми файлами, без
компиляции. **Не ставьте на сервер Python 3.14.**

> 🔑 **Самое важное правило для офлайна:** домашняя и рабочая виртуалки должны
> быть **одной и той же версии Ubuntu** (например, обе 24.04 LTS). Тогда пакеты,
> скачанные дома, без проблем встанут на работе.

## ЧАСТЬ A. Создаём виртуальную машину (Hyper-V) и ставим Ubuntu

### A1. Включить Hyper-V (если ещё не включён)
1. «Панель управления» → «Программы» → «Включение/отключение компонентов
   Windows» → поставьте галочку **Hyper-V** → ОК → перезагрузка.

### A2. Скачать образ Ubuntu Server
1. На сайте https://ubuntu.com/download/server скачайте **Ubuntu Server 24.04
   LTS** (файл `ubuntu-24.04.x-live-server-amd64.iso`).
2. Сохраните `.iso` — он же потом поедет на флешку для работы.

### A3. Настроить сеть Hyper-V (один раз)
1. Откройте **«Диспетчер Hyper-V»** → справа **«Диспетчер виртуальных
   коммутаторов»** → **«Создать виртуальный коммутатор»** → тип **«Внешний»
   (External)** → выберите вашу сетевую карту → имя, например `ExternalSwitch`
   → ОК. Это даст виртуалке доступ в сеть (и в интернет дома).

### A4. Создать виртуальную машину
1. «Диспетчер Hyper-V» → **«Создать» → «Виртуальная машина»**.
2. Имя: `servicedesk-home`. Поколение: **2 (Generation 2)**.
3. Память: **2048–4096 МБ**. Сеть: выберите `ExternalSwitch`.
4. Диск: новый, **20–30 ГБ**.
5. Установка: «Установить ОС с файла образа» → укажите скачанный `.iso`.
6. После создания: ПКМ по ВМ → «Параметры» → **«Безопасность»** → СНИМИТЕ
   галочку **«Включить безопасную загрузку» (Secure Boot)** *или* выберите
   шаблон **«Microsoft UEFI Certificate Authority»** (иначе Ubuntu может не
   загрузиться). Сохраните.

### A5. Установить Ubuntu Server
1. Запустите ВМ («Подключить» → «Пуск»), загрузится установщик Ubuntu.
2. Идите по шагам (язык, раскладка). На вопросах сети — оставьте автоматически
   (DHCP). 
3. **«Profile setup»**: придумайте имя пользователя (например `admin`) и
   пароль — запомните их.
4. **Обязательно** отметьте пункт **«Install OpenSSH server»** — это позволит
   подключаться к виртуалке по сети и удобно копировать файлы.
5. Дойдите до конца, дождитесь установки, **«Reboot Now»**.
6. После перезагрузки войдите под своим пользователем (логин/пароль из п.3).

### A6. Узнать IP-адрес виртуалки
В виртуалке выполните:
```bash
ip a
```
Найдите строку вида `inet 192.168.1.50/24` — это IP вашей ВМ (запишите).
Дальше можно работать прямо в окне ВМ или подключиться с Windows по SSH:
```cmd
ssh admin@192.168.1.50
```

> 💡 Все длинные команды удобнее копировать через SSH из Windows-терминала,
> чем набирать в окне Hyper-V вручную.

## ЧАСТЬ B. Запуск ServiceDesk дома (без домена)

Здесь у вас дома есть интернет, поэтому ставим всё обычным способом. Заодно
поймёте, что и зачем устанавливается, — на работе сделаем то же самое офлайн.

### B1. Обновить систему и поставить базовые пакеты
```bash
sudo apt-get update
sudo apt-get install -y python3 python3-venv python3-pip apache2
```
- `python3-venv` — для изолированного окружения проекта;
- `python3-pip` — установщик Python-библиотек;
- `apache2` — веб-сервер (обратный прокси; на работе он же будет делать Kerberos).

### B2. Скопировать код проекта на виртуалку
Положим проект в `/opt/servicedesk`. Способы передать файлы:
- **По SSH с Windows** (рекомендуется): программой **WinSCP** или командой
  `scp` скопируйте папку проекта в домашнюю папку пользователя ВМ, затем:
  ```bash
  sudo mkdir -p /opt/servicedesk
  sudo cp -r ~/sd_main/* /opt/servicedesk/     # путь, куда скопировали
  ```
- Внутри `/opt/servicedesk` должен лежать `run.py` (в корне), рядом — папка
  с кодом, `requirements.txt`, `sync_ad_users.py`, `init_db.py` и т.д.

- sudo cp -r ~/ServiceDesk/* /opt/servicedesk/

Создадим системного пользователя, от имени которого будет работать сервис, и
отдадим ему папку:
```bash
sudo useradd --system --shell /usr/sbin/nologin servicedesk
sudo chown -R servicedesk:servicedesk /opt/servicedesk
```

### B3. Создать виртуальное окружение и поставить зависимости
```bash
cd /opt/servicedesk
sudo -u servicedesk python3 -m venv .venv
sudo -u servicedesk .venv/bin/pip install --upgrade pip
sudo -u servicedesk .venv/bin/pip install -r requirements.txt
```
> Все наши зависимости (Flask, ldap3, gunicorn, openpyxl…) — чистый Python,
> поэтому ставятся быстро и без компиляции.

### B4. Настроить .env
```bash
sudo -u servicedesk cp .env.example .env
sudo -u servicedesk nano .env
```
Дома заполните минимально:
- `SERCET_KEY` — длинная случайная строка. Сгенерировать:
  ```bash
  .venv/bin/python -c "import secrets; print(secrets.token_hex(32))"
  ```
- `DATABASE_URL=sqlite:///servicedesk.db` (дома проще всего SQLite);
- `KERBEROS_SSO_ENABLED=false` (дома домена нет → вход по логину/паролю);
- `BOOTSTRAP_ADMIN_EMAIL` и `BOOTSTRAP_ADMIN_PASSWORD` — чтобы создать
  локального админа для входа.
Сохраните (Ctrl+O, Enter, Ctrl+X).

### B5. Создать базу и (по желанию) тестовые данные
Создать таблицы и запасного админа:
```bash
sudo -u servicedesk .venv/bin/python init_db.py
```
Либо, чтобы посмотреть приложение с тестовыми пользователями/заявками, можно
один раз залить демо-данные (ВНИМАНИЕ: стирает базу):
```bash
sudo -u servicedesk .venv/bin/python seed_data.py
```

### B6. Быстрая проверка «вживую» (без Apache)
Запустим напрямую gunicorn, чтобы убедиться, что приложение поднимается:
```bash
sudo -u servicedesk .venv/bin/gunicorn -k gthread -w 1 --threads 8 -b 0.0.0.0:8000 run:app
```
Откройте в браузере на любом домашнем ПК: `http://<IP-виртуалки>:8000`
Должна открыться страница входа. Войдите локальным админом (из .env).
Проверили — нажмите `Ctrl+C`, чтобы остановить.

### B7. Сделать автозапуск через systemd
1. Скопируйте готовый юнит из проекта и поправьте при необходимости пути:
   ```bash
   sudo cp /opt/servicedesk/deploy/servicedesk.service /etc/systemd/system/servicedesk.service
   sudo nano /etc/systemd/system/servicedesk.service   # проверьте пути/пользователя
   ```
2. Включите и запустите сервис:
   ```bash
   sudo systemctl daemon-reload
   sudo systemctl enable --now servicedesk
   sudo systemctl status servicedesk        # должно быть "active (running)"
   ```
   Логи приложения смотреть так:
   ```bash
   sudo journalctl -u servicedesk -f
   ```

### B8. Поставить Apache «перед» приложением (порт 80)
1. Включите нужные модули и наш сайт:
   ```bash
   sudo a2enmod proxy proxy_http headers
   sudo cp /opt/servicedesk/deploy/apache-servicedesk.conf /etc/apache2/sites-available/servicedesk.conf
   sudo a2dissite 000-default          # выключить дефолтный сайт
   sudo a2ensite servicedesk
   sudo systemctl restart apache2
   ```
2. Чтобы открывать по имени, на домашнем ПК (Windows) добавьте в файл
   `C:\Windows\System32\drivers\etc\hosts` строку:
   ```
   192.168.1.50   servicedesk.local
   ```
   (IP — вашей виртуалки). Теперь сервис открывается по `http://servicedesk.local`.

**Готово — дома всё работает.** Вы увидели полный цикл: код → venv → база →
gunicorn(systemd) → Apache. На работе будет то же самое, только пакеты — офлайн,
и добавится домен/Kerberos.

## ЧАСТЬ C. Готовим «офлайн-комплект» на флешку (дома)

Цель — скачать дома ВСЁ, что понадобится на работе без интернета. Делать это
нужно на **такой же** виртуалке Ubuntu (та же версия!), что и на работе.

Создадим папку комплекта:
```bash
mkdir -p ~/offline-bundle/deb ~/offline-bundle/wheels
```

### C1. Скачать системные пакеты (.deb) вместе с зависимостями
Скачиваем (но НЕ устанавливаем) всё, что нужно на работе, включая пакеты для
домена/Kerberos:
```bash
cd ~/offline-bundle/deb
sudo apt-get update
sudo apt-get install --download-only --reinstall -y \
  python3 python3-venv python3-pip \
  apache2 libapache2-mod-auth-gssapi \
  krb5-user sssd sssd-tools realmd adcli \
  samba-common-bin oddjob oddjob-mkhomedir packagekit libnss-sss libpam-sss
sudo cp /var/cache/apt/archives/*.deb ~/offline-bundle/deb/
sudo chown -R $USER:$USER ~/offline-bundle/deb
```
- `--download-only` — только скачать;
- `--reinstall` — заставляет скачать пакеты, даже если они уже стоят на этой ВМ
  (иначе на работе может не хватить какого-то «уже установленного дома» пакета).

> 🔎 Важный нюанс офлайна: набор скачанного зависит от того, что УЖЕ стоит в
> системе. Поэтому идеально — качать на **чистой** виртуалке, установленной с
> того же ISO, что и рабочая. Если домашняя и рабочая ВМ ставились из одного
> `.iso` — состав базовой системы совпадает, и пакетов хватит.

### C2. Скачать Python-библиотеки (wheels)
```bash
cd /opt/servicedesk
sudo -u servicedesk .venv/bin/pip download -r requirements.txt -d ~/offline-bundle/wheels
```
Если на работе другая СУБД (например PostgreSQL) — добавьте её драйвер:
```bash
.venv/bin/pip download psycopg2-binary -d ~/offline-bundle/wheels
```

### C3. Сложить в комплект всё остальное
Положите в `~/offline-bundle`:
- `ubuntu-24.04.x-live-server-amd64.iso` (тот же образ, что ставили дома);
- папку с **кодом проекта** (`sd_main` целиком);
- этот файл инструкции.

Скопируйте `~/offline-bundle` на флешку. Передать с виртуалки на Windows-хост
проще всего по SSH (WinSCP/scp), а уже с Windows — на флешку.

**Итог комплекта на флешке:**
```
offline-bundle/
├─ ubuntu-24.04.x-live-server-amd64.iso   (установка ОС на работе)
├─ deb/        (системные пакеты .deb)
├─ wheels/     (python-библиотеки)
└─ sd_main/    (код проекта)
```

## ЧАСТЬ D. Развёртывание на работе (в домене, офлайн)

### D1. Поставить виртуалку и ОС
То же, что Часть A, но `.iso` берёте с флешки. Поставьте Ubuntu Server **той же
версии**. На работе уточните у сисадмина статический IP, маску, шлюз, DNS —
**DNS обязательно должен указывать на контроллер домена**, иначе Kerberos и
вход в домен не заработают. **<<< УЗНАТЬ У СИСАДМИНА: IP/маска/шлюз/DNS >>>**

### D2. Скопировать комплект с флешки
Скопируйте папку `offline-bundle` с флешки на сервер (по SSH с рабочего ПК или
смонтировав флешку в гостя). Разложите проект:
```bash
sudo mkdir -p /opt/servicedesk
sudo cp -r ~/offline-bundle/sd_main/* /opt/servicedesk/
sudo useradd --system --shell /usr/sbin/nologin servicedesk
sudo chown -R servicedesk:servicedesk /opt/servicedesk
```

### D3. Установить системные пакеты офлайн
```bash
cd ~/offline-bundle/deb
sudo dpkg -i *.deb        # установит все .deb из папки
sudo apt-get install -f --no-download   # доустановит связи из этой же папки, без интернета
```
Если `dpkg` ругается на порядок зависимостей — повторите `sudo dpkg -i *.deb`
ещё раз (он доставит то, что не встало с первого прохода).

### D4. Поставить Python-библиотеки офлайн
```bash
cd /opt/servicedesk
sudo -u servicedesk python3 -m venv .venv
sudo -u servicedesk .venv/bin/pip install --no-index --find-links=$HOME/offline-bundle/wheels -r requirements.txt
```
`--no-index` запрещает выход в интернет, `--find-links` берёт пакеты из папки.

### D5. Настроить .env (боевой)
```bash
sudo -u servicedesk cp .env.example .env
sudo -u servicedesk nano .env
```
Заполните доменные значения (`AD_*`) и `KERBEROS_SSO_ENABLED=true`, `DATABASE_URL`
(SQLite или СУБД от сисадмина), `SERCET_KEY`. Все заглушки — из чек-листа
`docs/AD_KERBEROS_IIS_SETUP.md` (раздел 1) — они общие и для Linux.

### D6. Ввести сервер в домен
```bash
# Проверить, что домен виден:
realm discover company.local            # <<< домен сисадмина >>>
# Ввести в домен (попросит пароль учётки с правом ввода в домен):
sudo realm join --user=Администратор company.local
# Проверить:
realm list
```
> `sssd` (ставится из комплекта) свяжет Linux с AD; `realm join` создаст
> доменную «учётку компьютера». Если `realm discover` ничего не находит —
> почти всегда проблема в DNS (см. D1). **<<< согласовать с сисадмином >>>**

### D7. Создать keytab для веб-службы (Kerberos)
Чтобы Apache мог проверять Kerberos-билеты, нужен **keytab** службы `HTTP`
для имени сервера. Его обычно делает сисадмин на контроллере домена (команда
`ktpass`) или вы — через `adcli`. Имя службы: `HTTP/servicedesk.company.local`.
Файл keytab положите в `/etc/apache2/servicedesk.keytab` и дайте права Apache:
```bash
sudo chown root:www-data /etc/apache2/servicedesk.keytab
sudo chmod 640 /etc/apache2/servicedesk.keytab
```
> Точную процедуру и SPN (`HTTP/servicedesk.company.local`) согласуйте с
> сисадмином — у него есть права в AD. **<<< УЗНАТЬ У СИСАДМИНА: keytab/SPN >>>**

### D8. Запустить приложение и Apache (как дома)
```bash
sudo -u servicedesk .venv/bin/python init_db.py     # создать таблицы
sudo -u servicedesk .venv/bin/python sync_ad_users.py   # ПЕРВИЧНАЯ выгрузка всех пользователей AD
sudo cp deploy/servicedesk.service /etc/systemd/system/
sudo systemctl daemon-reload && sudo systemctl enable --now servicedesk

sudo a2enmod proxy proxy_http headers auth_gssapi
sudo cp deploy/apache-servicedesk.conf /etc/apache2/sites-available/servicedesk.conf
sudo nano /etc/apache2/sites-available/servicedesk.conf
#   -> впишите реальный ServerName
#   -> РАСКОММЕНТИРУЙТЕ блок <Location /> с Kerberos (AuthType GSSAPI ...)
sudo a2dissite 000-default && sudo a2ensite servicedesk
sudo systemctl restart apache2
```

### D9. Ночная синхронизация пользователей (cron)
Добавьте задачу, которая каждую ночь обновляет список пользователей:
```bash
sudo crontab -u servicedesk -e
```
и впишите строку (запуск в 03:00):
```
0 3 * * * cd /opt/servicedesk && /opt/servicedesk/.venv/bin/python sync_ad_users.py >> /opt/servicedesk/logs/cron.log 2>&1
```

### D10. Проверка Single Sign-On
На **доменном** рабочем ПК откройте `http://servicedesk.company.local`. Если
сайт добавлен в зону «Местная интрасеть» (это обычно делает групповая политика;
уточните у сисадмина), вход произойдёт автоматически, под вашей доменной
учёткой, без ввода пароля.

## Если что-то не работает (быстрая диагностика)

| Симптом | Что проверить |
|---------|---------------|
| Сайт не открывается совсем | `sudo systemctl status servicedesk` и `apache2`; firewall: `sudo ufw allow 80,8000/tcp` |
| 502 Bad Gateway от Apache | Не запущен gunicorn/servicedesk — смотрите `journalctl -u servicedesk -f` |
| `realm discover` ничего не находит | DNS не указывает на контроллер домена (D1); время на сервере должно совпадать с доменом (Kerberos чувствителен ко времени) — `timedatectl`, синхронизация NTP |
| Браузер просит логин/пароль (Kerberos не сработал) | Неверный SPN/keytab; сайт не в зоне «Местная интрасеть»; рассинхрон времени |
| Вход прошёл, но пользователь не распознан | Не запускали `sync_ad_users.py`, либо его логин не совпадает с `REMOTE_USER`; смотрите `logs/ad_sync.log` |
| Офлайн `dpkg -i` жалуется на зависимости | В комплекте не хватает какого-то .deb — доскачайте дома на ЧИСТОЙ ВМ той же версии (Часть C, примечание про `--reinstall` и чистую систему) |
| Чат в заявках «живёт» через long-polling, не websocket | Так и задумано (gthread). Для внутреннего сервиса этого достаточно. |

## Короткая шпаргалка

**Дома:**
1. Hyper-V → ВМ → Ubuntu Server 24.04 (+OpenSSH).
2. `apt install python3-venv python3-pip apache2`.
3. Код в `/opt/servicedesk`, `venv`, `pip install -r requirements.txt`.
4. `.env` (SQLite, SSO=false, локальный админ) → `init_db.py`.
5. `gunicorn ... run:app` для проверки → systemd → Apache.
